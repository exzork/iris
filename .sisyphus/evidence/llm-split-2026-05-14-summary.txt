### Backward Compatibility

- Existing operators using only LLM_TIMEOUT will get:
  - Chat timeout: 2m (up from 30s)
  - Tool timeout: LLM_TIMEOUT * 4 (rounded up to minute)
- New operators can set LLM_CHAT_TIMEOUT and LLM_TOOL_TIMEOUT independently
- Lore LLM behavior unchanged (uses cfg.LoreLLMTimeout with context.WithTimeout)

### Root Cause Resolution

The original issue was that a single `chatClient` with 30s timeout was used for all LLM calls:
- Plain chat replies: 30s (too short for complex responses)
- Tool-call streams: 30s (too short for multi-round tool execution)

Solution splits the timeout:
- Chat: 2m (sufficient for most replies)
- Tools: 10m (sufficient for multi-round tool execution with streaming)

This allows tool-call streams to complete without hitting the deadline mid-body.
