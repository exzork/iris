Task T2: ChatWithTools Implementation - Tool-Call Loop
========================================================

DELIVERABLES COMPLETED
======================

1. New Public Types & Method in internal/llm/client.go
   - ToolExecutor interface: Execute(ctx context.Context, name string, args map[string]interface{}) (string, error)
   - ChatWithToolsConfig struct with fields:
     * Model: string
     * GuildID: int64
     * Tools: []map[string]interface{} (OpenAI-shaped)
     * Exec: ToolExecutor
     * Max: int (default 3 if 0)
   - ChatWithTools(ctx context.Context, messages []map[string]string, cfg ChatWithToolsConfig) (string, error)

2. Tool-Call Loop Implementation
   - Per-round logic:
     1. Build request with model, messages, temperature/max_tokens from client config
     2. Include tools when len(cfg.Tools) > 0
     3. POST /v1/chat/completions with stream=false
     4. Parse response
     5. If no tool_calls, return Choices[0].Message.Content
     6. Else for each tool_call:
        - Append assistant message with role=assistant and original tool_calls array
        - Call cfg.Exec.Execute(ctx, name, args)
        - On error, use "error: "+err.Error() as tool content
        - Append role=tool message with tool_call_id and content
     7. round++; if round >= cfg.Max, return last assistant content or "", nil
     8. Loop continues
   - Message history: kept as []map[string]string in public signature
   - Internal enrichment: []map[string]interface{} for request body with tool turns
   - Audit metadata: attached via llm.WithMeta(ctx, &llm.ContextMeta{TriggerReason: "chat_with_tools", GuildID: cfg.GuildID})
   - DEBUG logging per round: tool_call round=N tool=<name> args=<short>
   - Respects ctx cancellation and timeout
   - Uses existing c.http, c.config.APIKey, retry behavior

3. Test Coverage (internal/llm/client_tools_test.go)
   - TestChatWithTools_NoTools_FallsThroughToText
     * cfg.Tools empty, server returns text
     * Verifies no tools in request
     * Returns text response
   
   - TestChatWithTools_OneToolCall_ThenText
     * First request: server returns tool_call
     * Second request: verifies role=tool message with matching tool_call_id and content
     * fakeToolExecutor records name+args
     * Final return equals second response's text
   
   - TestChatWithTools_MaxRounds_Exceeded_ReturnsLastText
     * Server always returns tool_calls
     * Loop aborts after Max rounds (set to 2)
     * Returns last assistant content (non-erroring)
     * Exactly 2 requests made
   
   - TestChatWithTools_ToolExecuteError_PropagatedAsToolMessage
     * Executor returns error
     * Second POST body carries content: "error: tool failed"
     * No panic, loop continues
     * Final response from model after error
