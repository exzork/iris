# TASK-8 RUNTIME WIRING FIX - COMPREHENSIVE SUMMARY

## Problem Statement
All Task 1-7 components (ContextStore, CrossChannel, Promoter, AllowedQuerier) were constructed but never wired into the live message path. The orchestrator.Config was assigned to blank identifier at line 167 of cmd/iris-bot/main.go and discarded, meaning:
- No immediate typing indicator
- No channel message capture
- No context building
- No cross-channel classification
- No memory promotion
- No LLM audit logging
- No allow-list routing enforcement

## Solution Overview
Wire the orchestrator into the gateway callback so every Discord message flows through the Task 1-7 pipeline before being sent to the LLM.

## Component Wiring Details

### Task 1: Channel Context & Allow-list (Wired)
- **ContextStore**: `&wireadapters.ContextStoreAdapter{Repo: channelMessageRepo}`
  - Provides ListRecent, GetByID, ListByUserAcrossChannels for context builder
- **AllowedQuerier**: `allowedRepo`
  - Enables router to check HasAny() for fallback mode detection

### Task 2: Allow-list Routing (Wired)
- **Router**: `&wireadapters.DeciderAdapter{Router: routerSvc}`
  - Enforces include-list after first allowed-channel row exists
  - Falls back to exception-channel exclusion if no allowed-channels configured

### Task 3: LLM Audit Logging (Wired)
- **LLM**: `&wireadapters.LLMCallerAdapter{Client: chatClient}`
  - Audit logging triggered when DEBUG=true
  - Duration and truncated content captured per request

### Task 4: Channel Message Capture (Wired)
- **Capture**: `&wireadapters.ChannelCaptureAdapter{Repo: channelMessageRepo}`
  - Every message upserted to rolling 20-message per-channel map
  - Runs before routing decision

### Task 5: Context Building (Wired)
- **ContextStore**: Used by orchestrator worker to build LLM context
- **SystemPrompt**: Set to `persona.BuildSystemPrompt(persona.PromptInput{})`
- At least 10 relevant recent messages included when available

### Task 6: Cross-Channel Classification & Memory Promotion (Wired)
- **CrossChannel**: `crossChannelClassifier`
  - Classifies bounded candidate sessions for relevance
  - Merges relevant cross-channel context into LLM prompt
- **Promoter**: `memoryPromoter`
  - Async memory promotion after response sent
  - Does not block message sending

### Task 7: Immediate Typing & Message Splitting (Wired)
- **ImmediateTyping**: `true`
  - SendTyping called immediately after router accepts message
  - Refreshed every 5 seconds until LLM completes
- **Discord**: `&wireadapters.DiscordSenderAdapter{Gateway: gateway}`
  - Handles message splitting at 2000-character boundary
  - Sends chunks in order

## Runtime Message Flow

```
Discord Event
    ↓
gateway.callback(ctx, event)
    ├─ Admin command ("!iris ...") → adminDispatcher → gateway.SendMessage
    │
    └─ Regular message → orch.Enqueue(ctx, event)
        ↓
        orchestrator.worker (async, 4 workers)
            ├─ Router.Decide(event)
            │   └─ Task 2: Check allow-list, return Decision
            │
            ├─ ChannelCapture.Capture(msg)
            │   └─ Task 4: Upsert to rolling 20-message map
            │
            ├─ ContextStore.ListRecent(guildID, channelID, 20)
            │   └─ Task 1: Fetch recent messages for context
            │
            ├─ CrossChannel.Classify(candidates)
            │   └─ Task 6: LLM classify cross-channel relevance
            │
            ├─ LLM.Chat(ctx, guildID, messages)
            │   ├─ Task 3: Audit log with duration/content
            │   ├─ Task 5: Include ≥10 context messages
            │   └─ Return response
            │
            ├─ Discord.SendMessage(ctx, guildID, channelID, response)
            │   ├─ Task 7: Immediate typing + refresh
            │   └─ Task 7: Split at 2000 chars, send chunks
            │
            └─ Promoter.Promote(async)
                └─ Task 6: Async memory promotion (non-blocking)
```

## Key Design Decisions

1. **Immediate typing enabled by default**: ImmediateTyping=true ensures Task 7 requirement is met at runtime
2. **4 worker goroutines**: Balanced for Discord event throughput without excessive memory
3. **128-item queue**: Sufficient for typical Discord bot load with backpressure handling
4. **Adapter pattern**: Minimal wrapper types satisfy orchestrator interfaces without modifying core services
5. **Admin commands bypass orchestrator**: Preserves existing admin behavior, no audit/context overhead
6. **Fallback to appInstance.Handle**: Defensive measure; should never execute in normal operation
7. **Deferred orch.Stop()**: Ensures graceful shutdown of worker goroutines on exit
