### FIX 1: Guild FK Constraint Resolution

**Problem**: FK violation on channel_conversations when guild 687347156524204067 was never inserted into guilds table. All guild-FK writes fail silently except lock_refresh which logs the error.

**Solution**: 
- Added `EnsureGuild(ctx, guildID)` method to `*GuildRepo` using `INSERT ... ON CONFLICT (id) DO NOTHING`
- Added `GuildEnsurer` interface and `GuildEnsurerAdapter` in wire/adapters.go
- Integrated guild ensure into `ChannelCaptureAdapter.Capture()` before `repo.Upsert(msg)`
- Added safety-net guild ensure call in gateway callback in main.go BEFORE `orch.Enqueue(ctx, event)`

**Files Changed**:
- `internal/repository/guild.go` (lines 40-48): Added EnsureGuild method
- `internal/app/wire/adapters.go` (lines 181-230): Added GuildEnsurer interface and updated ChannelCaptureAdapter
- `cmd/iris-bot/main.go` (lines 242-250): Added guild ensure in gateway callback
- `internal/repository/repository_test.go` (lines 41-68): Added TestGuildRepo_EnsureIsIdempotent test

**Test**: `TestGuildRepo_EnsureIsIdempotent` verifies idempotency - calling Ensure twice with same ID returns nil both times and does not duplicate rows.

### FIX 2: Memory Promoter Detached Context

**Problem**: Memory promoter uses per-job context which is canceled on handle() return, causing "context canceled" WARN logs.

**Solution**:
- Spawned promoter.Consider() in its own goroutine with detached context (15s timeout from context.Background())
- Tracked via existing refreshWG WaitGroup so Stop() drains promoter goroutines cleanly
- Promoter now completes independently of per-job context lifecycle

**Files Changed**:
- `internal/orchestrator/orchestrator.go` (lines 347-355): Wrapped promoter.Consider in detached goroutine with refreshWG tracking
- `internal/orchestrator/orchestrator_test.go` (lines 50-95): Added TestPromoterRunsOnDetachedContext test

**Test**: `TestPromoterRunsOnDetachedContext` verifies promoter context remains alive for 100ms after handle() returns, completing without "context canceled" error.

### FIX 3: Log Actual LLM Model

**Problem**: llm_request log hardcodes "gpt-4o-mini" but bot uses kr/claude-opus-4.7 via tier routing.

**Solution**:
- Added `TierRouter` field to orchestrator.Config
- Before Chat call, classify query tier and resolve actual model via TierRouter.ModelFor()
- Log resolved model name in llm_request instead of hardcoded value
- Falls back to "unknown" if TierRouter not configured

**Files Changed**:
- `internal/orchestrator/orchestrator.go` (lines 38-59): Added TierRouter to Config
- `internal/orchestrator/orchestrator.go` (lines 313-330): Determine and log actual model before Chat
- `cmd/iris-bot/main.go` (line 324): Wire tierRouter into orchCfg
- `internal/orchestrator/orchestrator_test.go` (lines 97-145): Added TestLLMRequestLogsActualModel test

**Test**: `TestLLMRequestLogsActualModel` verifies tier classification and model resolution work correctly.
